# E6 Issue 44: Generate Cabinet List

Status: Done
Issue: #44
Epic: #38

## Problem

Measured layouts, zones, and obstacles are useful only after Studio can turn
them into a first deterministic furniture module list. That generated list is
the handoff point for review, quotation, BOM, and Fusion 360 export.

## Goal

Generate an initial cabinet list from the saved measured layout using backend
rules, persist the result to the design session, and expose it in the layout UI.

## Acceptance Criteria

- [x] Cabinet list is generated from measured layout, obstacles, and cabinet templates.
- [x] Each cabinet has name/type, width, height, depth, wall/segment, approximate position, and notes.
- [x] Total cabinet width is validated against available wall length.
- [x] Appliances and required clearance zones are respected.
- [x] If modules do not fit, the API returns actionable warnings.
- [x] Generated cabinet list is saved to the design session.

## Implementation Notes

- `POST /api/design-sessions/{sessionId}/cabinet-plan` generates and saves the
  cabinet plan.
- `GET /api/design-sessions/{sessionId}/cabinet-plan` reopens the latest saved
  plan snapshot.
- Studio uses the saved `measured-layout.json` as the source of truth.
- Base cabinets are generated by wall segment, reserving measured functional
  zones first.
- Obstacles block base cabinets only when they overlap the base-cabinet height
  band. A window with `zMm` above the base height does not block lower cabinets.
- The generated list is persisted as `Cabinet` records attached to a
  `KitchenSpec`, and as an inline `cabinet-plan.json` artifact for exact reopen.
- Active `CabinetTemplate` records are used when they have usable dimensions;
  otherwise deterministic fallback template codes are used and can be replaced
  by seeded production templates later.

## API

```http
POST /api/design-sessions/{sessionId}/cabinet-plan
GET /api/design-sessions/{sessionId}/cabinet-plan
```

Both endpoints require authentication.
